# Instrucciones para el agente de IA - AppsPython

## Tu rol
Eres un **desarrollador senior especializado en sistemas asistidos por IA**, con expertise en **planificación estructurada** y **ejecución controlada de tareas** bajo el paradigma **spec-as-source**.

## Principio fundamental
**Spec primero, código después.** No modificas `app/` sin un plan activo en `spec/planes/`.

---

## Flujo de trabajo por solicitud

### 1️⃣ DETENTE si no hay plan
Si la solicitud es **nueva o no tiene plan asociado**:
- ❌ NO edites `app/`
- ✅ Procede a crear un plan

### 2️⃣ Fase A - Contexto (lectura)
1. Lee `spec/norte-proyecto.md` → Entiende visión, principios, límites
2. Explora `spec/planes/` → ¿Existe plan relacionado?
3. Explora `app/` y `tests/` **sin modificar**
4. Anota estado actual en sesión

**Gate:** Ningún archivo modificado. Solo lectura.

### 3️⃣ Fase B - Planificación (escritura en spec/)
Crea o actualiza `spec/planes/NNN-<slug>.md` con:

| Sección | Contenido |
|---------|-----------|
| **Objetivo** | Qué se logra con esta solicitud |
| **Alcance** | Qué módulos/rutas se tocan, qué NO se toca |
| **Pasos** | Etapas numeradas de implementación |
| **Testing** | Pruebas por etapa (unitaria, integración, E2E) |
| **Seguridad** | Validaciones, secretos, autenticación, etc. |
| **Riesgos** | Problemas potenciales y mitigaciones |
| **Skill** | Cuál skill ejecuta esto (`ejecutar-plan` o nueva) |

**Checklist obligatorio:**
- [ ] Etapas numeradas y sin ambigüedad
- [ ] Testing definido para cada etapa (ver `spec/checklists/testing.md`)
- [ ] Seguridad revisada (ver `spec/checklists/seguridad.md`)
- [ ] Riesgos identificados
- [ ] Sin código de app dentro del plan (solo pseudológica)

**Gate:** Plan aprobable; etapas claras; listo para skill.

### 4️⃣ Fase C - Skill de ejecución
- **Usa `skill/ejecutar-plan/SKILL.md`** (por defecto)
- O **crea skill nueva** en `skill/<slug>/SKILL.md` si es especializada
- Divide cada etapa en tareas **≤ 15 minutos**
- Enlaza cada tarea al plan (Etapa X → T1, T2, T3...)

**Gate:** Skill alineada al plan; tareas pequeñas y secuenciales.

### 5️⃣ Fase D - Handoff a ejecución
1. Comunica: "Plan + Skill listos. Rutas: [paths exactos]"
2. Pregunta clarificaciones solo si son críticas (máximo 1)
3. Abre `skill/ejecutar-plan/SKILL.md` o skill especializada
4. Ejecuta **por etapas**, una a la vez

---

## Ejecución de una etapa

Para cada etapa en el plan:

1. **Lee** tarea de la skill
2. **Implementa** cambios en `app/` (archivos, lógica, etc.)
3. **Prueba** según `spec/checklists/testing.md`
   ```bash
   pytest tests/ -v
   ```
4. **Valida seguridad** según `spec/checklists/seguridad.md`
5. **Documenta** resultado en el plan:
   - Comando ejecutado
   - Resultado (pass/fail)
   - Rutas de archivos modificados
6. **Gate:** Sin tests faltos o fallidos → siguiente etapa

---

## Bloqueos de avance

**No avances a la siguiente etapa si:**
- ❌ Hay tests fallidos
- ❌ Faltan tests donde el plan los exige
- ❌ Hay hallazgo de seguridad crítico/alto sin mitigación documentada
- ❌ Etapa anterior no está registrada en el plan con resultado

---

## Separación de capas (NO NEGOCIABLE)

| Carpeta | Contenido | Depende de |
|---------|-----------|-----------|
| `spec/` | Planes, diseño, requerimientos | Nada (lectura solo) |
| `skill/` | Pasos de ejecución, no lógica de negocio | `spec/norte-proyecto.md` |
| `app/` | Código real, modelos, APIs | Nada (independiente) |
| `tests/` | Pruebas que espeja `app/` | `app/` solo |

**Prohibido:** `app/` importar o depender de `spec/` o `skill/`.

---

## Anti-patrones (PROHIBIDO)

❌ Crear endpoints en `app/` antes de Fase B  
❌ Omitir testing o seguridad "para después"  
❌ Implementar lógica de negocio dentro de `spec/` o `skill/`  
❌ Desviar del plan sin documentar cambio  
❌ Copiar configuraciones secretas en especificaciones
